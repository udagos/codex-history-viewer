# Plan: Folders Pin/Tag & View Sorting

## Overview
Based on user requirements, the extension will be updated to:
1. Allow users to pin and tag folders as independent entities (not just their sessions).
2. Separate the "Pinned" view into two collapsible groups: "Folders" and "Sessions".
3. Add flexible sorting functions to both the "Pinned" view and the "History" view.

## Step-by-Step Implementation

### 1. Update Core Data Models for Folder Nodes
- **`src/tree/treeNodes.ts`**: 
  - Add new node types for grouping in the Pinned view: `PinnedFoldersGroupNode` and `PinnedSessionsGroupNode`.
  - Update `toTreeItemContextValue` so context values (`codexHistoryViewer.folder` and `codexHistoryViewer.folderPinned`) are handled correctly.

### 2. Enable Folder Pinning and Tagging in Commands
- **`src/extension.ts`**:
  - Add a new helper `collectEntityFsPaths` (or update existing path collectors) to return the `cwd` of a `FolderNode` directly instead of resolving it to its child sessions, specifically for actions like "Pin", "Unpin", and "Edit Tag/Annotation".
  - Ensure the `pinSession`, `unpinSession`, and `editSessionAnnotation` command handlers use this new helper to properly target the folder.
  - Leave `deleteSessions` and `exportSessions` resolving to child sessions so that deleting a folder still means deleting its contents.
- **`package.json`**:
  - Update the `when` clauses in `menus.view/item/context` for pinning/unpinning and tagging to also show up when `viewItem == codexHistoryViewer.folder` or `codexHistoryViewer.folderPinned`.

### 3. Pinned View Separate Groups & Sorting
- **`src/tree/pinnedTree.ts`**:
  - Update `PinnedTreeDataProvider.getChildren()` to return two top-level groups: `PinnedFoldersGroupNode` and `PinnedSessionsGroupNode` (unless there are no pins at all, then show the drop hint).
  - Modify how `pins` are evaluated: check if the pinned `fsPath` exists in `this.historyService.getIndex().byFolder` to identify it as a folder pin. Otherwise, treat it as a session pin or a missing pin.
  - **Sorting**: Add a sort preference (e.g., `PinnedViewSortMode`: `date`, `name`, `pinnedAt`). Apply this sorting when rendering the children of the "Folders" and "Sessions" groups.

### 4. History View Sorting
- **`src/tree/historyTree.ts`**:
  - Introduce sorting configuration (e.g., `HistoryFolderSortMode`: `name`, `recentActivity`, etc.) for the folder view.
  - Apply the sorting logic before returning nodes in `getChildren()`.

### 5. UI Commands & Settings for Sorting
- **`package.json` & `src/extension.ts`**:
  - Add commands to change the sorting orders:
    - `codexHistoryViewer.ui.sortPinnedBy...`
    - `codexHistoryViewer.ui.sortHistoryFoldersBy...`
  - Add header menu actions (`view/title` inside `menus` in `package.json`) with an icon (e.g., `$(sort-precedence)`) to expose these sorting options via a dropdown or inline actions.
  - Save the selected sort modes into `workspaceState` so they persist across sessions.

## Considerations
- **Backward Compatibility**: Existing folder pins (if a user somehow managed to pin a folder path) or session pins will remain valid. `PinStore` only stores `fsPath` which is agnostic to files vs folders.
- **Empty Groups**: If there are no folders pinned, the "Folders" group can either be hidden or show an "Empty" placeholder. Hiding it when empty keeps the UI clean.
- **Performance**: Differentiating between folder and session using `this.historyService.getIndex().byFolder.has(p.fsPath)` is an O(1) synchronous operation and keeps `getChildren` fast.
